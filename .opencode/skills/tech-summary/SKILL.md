# 技术内容深度分析技能

## 使用场景

当需要对采集的 GitHub Trending、Hacker News 等技术内容进行深度分析和总结时使用此技能。适用于知识库系统的分析环节，从原始数据中提取洞察、发现趋势，为知识整理和分发提供高质量分析结果。

## 执行步骤

### 1. 读取最新采集文件
使用 Glob 和 Read 工具查找 `knowledge/raw/` 目录下最新的采集文件（按文件名中的日期排序），读取 JSON 格式的原始数据。优先处理 `github-trending-YYYY-MM-DD.json` 文件，确保数据新鲜度。

### 2. 逐条深度分析
对原始数据中的每个项目进行深度分析，生成结构化分析结果：

- **摘要**：不超过 50 字的中文摘要，精炼概括项目核心价值
- **技术亮点**：2-3 个具体的技术亮点，用事实和数据说话
  - 示例："采用 Rust 重写核心模块，性能提升 40%"
  - 示例："引入新的注意力机制，在基准测试中准确率提高 15%"
- **评分**：1-10 分整数评分，必须附详细评分理由
- **标签建议**：3-5 个相关标签，用于分类和检索

### 3. 趋势发现
基于所有项目的分析结果，识别整体趋势和模式：

- **共同主题**：多个项目中反复出现的技术主题或方向
- **新概念**：首次出现或正在兴起的新技术概念、框架、方法论
- **技术热点**：当前最受关注的技术领域和工具栈
- **潜在机会**：分析结果中暗示的技术发展机会或空白领域

### 4. 输出分析结果
将分析结果保存为 JSON 格式文件，包含原始数据、深度分析和趋势发现三个层次的信息。

## 评分标准

采用 10 分制评分体系，严格遵循以下标准：

- **9-10 分（改变格局）**：具有颠覆性创新，可能改变技术格局或开辟新方向
  - 示例：全新的 AI 架构、革命性的编程范式、解决长期痛点的新方案
  - 约束：15 个项目中 9-10 分项目不超过 2 个

- **7-8 分（直接有帮助）**：对开发者有直接实用价值，能显著提升工作效率
  - 示例：优秀的工具库、实用的框架、清晰的教程、高效的开发工具

- **5-6 分（值得了解）**：有一定技术价值，值得了解和关注
  - 示例：技术改进、优化方案、有趣的概念验证

- **1-4 分（可略过）**：技术价值有限，可以略过或仅作参考
  - 示例：重复造轮子、质量一般的项目、过时的技术方案

**评分原则**：
1. 客观公正，基于事实和数据评分
2. 评分理由必须具体、可验证
3. 考虑项目的创新性、实用性、影响力、代码质量
4. 避免个人偏好，保持技术中立

## 注意事项

1. **数据完整性**：确保分析覆盖原始数据中的所有项目，不遗漏任何条目
2. **分析深度**：技术亮点必须基于具体事实，避免空泛描述
3. **趋势客观性**：趋势发现应基于数据分析，而非主观臆测
4. **评分严谨性**：严格遵守评分标准和约束条件，保持评分一致性
5. **时效性**：优先分析最新数据，确保洞察的时效价值

## 输出格式

输出文件为 JSON 格式，结构如下：

```json
{
  "source": "原始数据来源，如 github_trending",
  "skill": "tech-summary",
  "analyzed_at": "2024-01-01T14:30:00Z",
  "original_file": "github-trending-2024-01-01.json",
  "items": [
    {
      "name": "项目名称",
      "url": "项目URL",
      "summary": "不超过50字的中文摘要",
      "technical_highlights": [
        "技术亮点1（具体事实）",
        "技术亮点2（具体事实）",
        "技术亮点3（具体事实）"
      ],
      "score": 8,
      "score_reason": "详细的评分理由，解释为什么给这个分数",
      "suggested_tags": ["tag1", "tag2", "tag3", "tag4", "tag5"]
    }
  ],
  "trend_analysis": {
    "common_themes": ["共同主题1", "共同主题2", "共同主题3"],
    "new_concepts": ["新概念1", "新概念2"],
    "technical_hotspots": ["热点领域1", "热点领域2"],
    "potential_opportunities": ["潜在机会1", "潜在机会2"]
  },
  "statistics": {
    "total_items": 15,
    "score_distribution": {
      "9_10": 2,
      "7_8": 5,
      "5_6": 6,
      "1_4": 2
    },
    "top_tags": ["llm", "agent", "framework", "rust", "python"]
  }
}
```

**字段说明**：
- `source`: 原始数据来源，从原始文件读取
- `skill`: 固定为 "tech-summary"，表示使用的技能名称
- `analyzed_at`: ISO 8601 格式的分析时间戳（UTC）
- `original_file`: 原始数据文件名
- `items`: 分析结果数组，每个元素对应一个项目的深度分析
  - `name`: 项目名称
  - `url`: 项目 URL
  - `summary`: 不超过 50 字的中文摘要
  - `technical_highlights`: 技术亮点数组（2-3 个）
  - `score`: 1-10 分整数评分
  - `score_reason`: 详细的评分理由
  - `suggested_tags`: 标签建议数组（3-5 个）
- `trend_analysis`: 趋势分析结果
  - `common_themes`: 共同主题数组
  - `new_concepts`: 新概念数组
  - `technical_hotspots`: 技术热点数组
  - `potential_opportunities`: 潜在机会数组
- `statistics`: 统计分析
  - `total_items`: 总项目数
  - `score_distribution`: 分数分布统计
  - `top_tags`: 最常出现的标签（按频率排序）

**文件命名**：`tech-summary-YYYY-MM-DD.json`，例如 `tech-summary-2024-01-01.json`
**保存路径**：`knowledge/articles/YYYY-MM/` 目录下，按月组织

## 验证要求

1. 每次运行后检查输出文件是否存在且格式正确
2. 确保所有项目的摘要不超过 50 字
3. 验证每个项目都有 2-3 个具体的技术亮点
4. 确认评分符合标准且 9-10 分项目不超过 2 个（针对 15 个项目）
5. 检查评分理由是否具体、可验证
6. 验证趋势分析基于数据，而非主观猜测
7. 确保输出文件保存到正确的目录结构
